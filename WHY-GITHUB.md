# なぜこの仕組みは GitHub の上に作られているのか

この仕組みにとって GitHub は単なる「ファイル置き場」ではない。
**記録・閲覧・蓄積・自動化の4つがひとつになった土台**である。共有フォルダやメール添付では、この4つは実現できない。

---

## 1. 報告書が「上書き」されず「育つ」 — 変更履歴

共有フォルダの Word ファイルは、誰かが上書きすれば前の姿は消える。
GitHub では**すべての変更が履歴（コミット）として残る**。いつ・誰が・どこを・なぜ変えたかを、いつでも遡れる。

だから、レポートに後から気づきを**追記して育てる**運用ができる。消える心配がないから、安心して書き換えられる。

## 2. 開いた瞬間に「読める」 — Markdown がそのままWebページになる

レポートは Markdown（.md）で書かれ、GitHub 上では**写真入りの報告書ページとしてそのまま表示される**。
閲覧する側に必要なのはブラウザだけ。Word も PowerPoint も開かない。

トップの README.md が自動更新される目次になっており、そこから全レポートへ1クリックで届く。
つまりリポジトリ全体が、そのまま**チームの報告書ポータルサイト**になる。

## 3. 報告書が「知識のネットワーク」になる — リンクによる蓄積

このリポジトリの核心はここにある。`archive-report` スキルがレポートから
**技術テーマ・企業情報・トレンド・アイデア**を抽出し、`KnowledgeBase/` に蓄積する。

- 企業ファイルから「その企業を見た出張レポート」へリンクで飛べる
- 技術ファイルには複数の出張で撮った写真が集まってくる
- 出張を重ねるほど、単発の報告書が**互いにつながった知識資産**へ変わる

ファイル同士を相対リンクでつなぎ、それがWeb上でクリックできる形で共有される。これは GitHub だからできることである。

## 4. AI の「作業場」ごと共有できる — 仕組みの再現性

レポートを自動生成する Claude Code のスキル（`.claude/commands/`）は、**リポジトリの中に同梱されている**。

- リポジトリを clone した人は、**データだけでなく自動化の仕組みごと**手に入れる
- 文体ルールや運用ルール（CLAUDE.md）も一緒に配られるので、誰が実行しても同じ品質になる
- スキル自体も版管理されるので、改善の履歴が残り、チームに展開できる

実際、このリポジトリ自体が山崎さんの Bishamon-Future-Product-Lab-OS を clone して仕組みごと移植したものであり、その再現性の実証になっている。

## 5. 安全に共同作業できる — 公開範囲と合流の仕組み

- **プライベート設定**で、招待した相手だけに公開できる
- 複数人で使う場合は、各自がブランチで書き、**Pull Request** で管理者が確認してから合流できる
- 感想・質問は **Issues** に残せば、それも記録として蓄積される

---

## まとめ：共有フォルダとの比較

| | 共有フォルダ + Word | GitHub + Markdown（この仕組み） |
|---|---|---|
| 変更履歴 | 上書きで消える | 全履歴が残り、追記で育つ |
| 閲覧 | ファイルを開く必要あり | ブラウザでそのまま読める |
| 知識の蓄積 | フォルダに溜まるだけ | リンクでつながる知識ベースになる |
| 自動化 | 人が手作業 | AIスキルがデータと一緒に配布される |
| 横展開 | 手順書を書いて説明 | clone すれば仕組みごと再現 |

**一言でいえば**：GitHub を使う意味は、出張報告を「書いて終わりの文書」から、**履歴を持ち、リンクでつながり、AIが世話をする、育ち続ける知識資産**に変えることにある。
