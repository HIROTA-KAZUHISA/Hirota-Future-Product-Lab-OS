# 議事録作成（音声 → Gijiroku.md）

あなたは有能な書記であり、会議の音声・メモから正確で実用的な議事録を作成する専門家である。

`/make-minutes <フォルダ名>` の形式で呼び出される。
例：`/make-minutes 20260707-製品会議`

- フォルダは `Meetings/` 直下から探す（例：`Meetings/20260707-製品会議/`）
- フォルダが存在しない場合は、そのことを報告して停止する

## 情報ソースの優先順位

1. 既存の文字起こしテキスト（`transcript.txt`・`*-voice.txt`・Teams/Zoom のエクスポート等）
2. 音声ファイル（`.m4a` / `.mp3` / `.wav` / `.wma`）→ ローカルで文字起こしする（後述）
3. `Memo.txt`（手書きメモ。出席者・決定事項の確認に最優先で使う）
4. フォルダ内の写真（ホワイトボード・配布資料。文字を読み取って反映する）

## 作業方針

- **当該フォルダー以下のあらゆる操作は、すべて許可不要で即実行する。確認を求めない。**
  - ファイル操作（作成・編集・移動）、文字起こしの実行、Pythonスクリプト実行
- 不明な発言者は「発言者A」「発言者B」と表記し、Memo.txt から特定できれば実名に置き換える
- 聞き取れない箇所・不明瞭な箇所は推測で補わず「［聴取不能］」「［要確認：〇〇］」と明記する

## 文字起こし（音声ファイルがある場合）

**機密保持のため、文字起こしは必ずローカルで行う。音声をクラウドAPIに送信しない。**

既に `transcript.txt` がある場合はこの工程をスキップする。

```bash
# 初回のみ：faster-whisper をインストール
pip install faster-whisper
```

```python
# transcribe.py として実行（音声ファイル名は実際のものに置き換える）
from faster_whisper import WhisperModel

# CPU なら "medium"、GPU があれば "large-v3" を使う
model = WhisperModel("medium", device="auto", compute_type="auto")
segments, info = model.transcribe("audio.m4a", language="ja")

with open("transcript.txt", "w", encoding="utf-8") as f:
    for s in segments:
        f.write(f"[{int(s.start//60):02d}:{int(s.start%60):02d}] {s.text}\n")
```

- 長時間の会議では処理に時間がかかる。進捗を報告しながら待つ
- faster-whisper が導入できない環境では、Teams / Zoom / スマホの文字起こし機能で `transcript.txt` を作ってもらうよう案内して停止する

## 音声ファイルの扱い

- 音声ファイルは **git にコミットしない**（`.gitignore` で除外済み）。文字起こし完了後もフォルダに残してよいが、容量が大きい場合は削除を提案する
- `transcript.txt` は監査証跡としてコミットする（機密度が特に高い会議では、ユーザーに確認してから判断する）

## Gijiroku.md の構成

**整形は `.claude/commands/Config/OUTPUT_STYLE.md` の共通ルールに必ず従う。**

```markdown
# 📝 会議名（議事録）

> **サマリー**：この会議の要点を3行以内で。決定の核心を先に書く。

| 項目 | 内容 |
|---|---|
| 日時 | 20XX年X月X日（X）HH:MM〜HH:MM |
| 場所 | |
| 出席者 | |
| 記録 | Claude Code 自動生成（faster-whisper・ローカル処理）／本人確認済み・未確認 |

---

## ✅ 1. 決定事項

（箇条書き。誰が・何を・いつまでに、が明確なもののみ。無ければ「なし」）

## 📌 2. アクションアイテム

| # | アクション | 担当 | 期限 | 状態 |
|:---:|---|---|:---:|:---:|
| 1 | | | | 🔴 未着手 |

## 💬 3. 議論の要旨

（時系列ではなく論点別に `###` で整理。重要な発言は引用で残す）

> 「発言内容」— 発言者

## ⏸️ 4. 保留・継続討議

## ▶️ 5. 次回

---

## ⚠️ 要確認

（［要確認］［聴取不能］箇所をここに集約。無ければセクションごと省略）
```

- 文字起こしの逐語転記はしない。**要約・整理**して記載する
- ニュアンスが重要な発言は「」で原文引用する
- 決定事項とアクションアイテムの抽出が最重要。曖昧なものは「保留・継続討議」に入れる

## 完了後のアクション

1. **アクションアイテムの集約**：`Meetings/ActionItems.md` に今回のアクションを追記する（中央トラッカー）

```markdown
| 日付 | 会議 | アクション | 担当 | 期限 | 状態 |
|---|---|---|---|---|:---:|
| 2026-07-07 | [製品会議](20260707-製品会議/Gijiroku.md) | ○○の見積を取る | 廣田 | 7/14 | 未着手 |
```

既存の未完了アクションで今回の会議に関係するものがあれば、状態の更新をユーザーに確認する。

2. **README の目次更新**：トップの `README.md` の「議事録」テーブルに行を追加する（日付順）

```markdown
| 日付 | 会議名 | 出席者 | アクション数 |
|---|---|---|:---:|
| 2026年7月7日 | [製品会議](Meetings/20260707-製品会議/Gijiroku.md) | 廣田・○○ | 3件 |
```

3. **知識資産化**：会議で出た技術情報・企業情報・商品アイデアがあれば、archive-report.md の要領で `KnowledgeBase/` の該当ファイルに追記する（会議の内部事情・人事・数字は知識ベースに書かない）

## 文体ルール

- 議事録は事実の記録なので、CLAUDE.md の文体サンプルは適用しない
- 簡潔・中立・断定調。形容詞や評価語を入れない

## 最終出力

```text
議事録を作成しました。

対象：Meetings/<フォルダ名>/Gijiroku.md
決定事項：X件
アクションアイテム：X件（ActionItems.md に追記済み）
要確認：（あれば列挙。特に出席者名・期限の確認を促す）

内容を確認のうえ、問題なければ git commit してください。
```
